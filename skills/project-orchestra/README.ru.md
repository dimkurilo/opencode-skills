# project-orchestra

🇬🇧 [English version](README.md)

Скилл для тех, кто крутит на одном продукте **несколько AI-агентов**: один ведёт, другой делает, третий проверяет. Чтобы папки и чаты не превращались в кашу.

Работает с OpenCode, Grok и другими CLI, которые читают `SKILL.md`.

**Версия:** 0.6.0 · **Лицензия:** MIT

---

## Мне это надо?

**Скорее да**, если:

- На одном сайте или продукте работают **два и больше** агента (или две разные модели).
- Ты режешь работу **кусками**: план, потом «ок, делаем», потом короткий отчёт. Не один вечный чат на всё.
- Темы лежат **отдельными подпапками** (оптимизация, лендинг, CRM, аудит), но **память проекта одна** - не «второй мозг» в каждой папке.
- Хочешь, чтобы «мы договорились» лежало **в файле**, а не только в старом чате.

**Скорее нет**, если:

- Нужна одна уже согласованная правка.
- Нужен дом для **одного** агента с длинными правилами - бери [project-bootstrap](../project-bootstrap/). Здесь для короткого старта есть четыре файла (`bootstrap-lite`).
- Ищешь SEO-методичку (title, скорость, контент). Этот скилл про **как агенты работают вместе**, не про SEO. SEO - другими скиллами, когда «офис» уже стоит.
- Хочется **новое имя скилла** на каждый тип задачи. Здесь наоборот: один вход - `/project-orchestra`.

## Зачем он

Без договорённостей обычно так:

| Что происходит | Чем кончается |
|----------------|---------------|
| Каждый раз заново: кто главный и куда класть файлы | Агенты затирают друг друга |
| «Договорились» осталось в чате | Через неделю никто не помнит, что утвердили |
| Два агента правят один статус | Каша в статусе и в памяти |
| Два `opencode run` параллельно «в лоб» | База лочится, странные ошибки |
| Отдельно помнишь wave-spec, bootstrap, orchestra | Три имени, один хаос |

Скилл даёт **общий язык папок** и простые правила: **кто что имеет право писать**.

### Откуда это

Не из учебника. Из практики: когда на одном продукте сидят разные CLI и модели, без лёгкой дисциплины всё разваливается.

Что вынесли:

- Сначала смотри, **какие инструменты реально есть** (CLI, скиллы, MCP), и уже от этого назначай роли. Самая «крутая» модель - ещё не начальник.
- Тяжёлую «комиссию» из нескольких моделей включай, когда область **новая**. Если уже делали похожее - не устраивай музей.
- Кусок работы: план, ревью, сделали, handoff. После «можно делать» лучше **новая сессия**, а не вечный чат.
- Важные выводы лучше проверить **другой** моделью, не той же семьёй, что писала.
- У продукта - **одна** оперативная память (HANDOFF / MEMORY). Темы - подпапки, не второй проект по умолчанию.

### Почему так, а не иначе

Сравнивали: всё в один толстый скилл, три-четыре разных имени, «только шаблоны без правил». Остановились на компромиссе: **одно имя**, внутри - темы, волны, проверка двумя моделями. wave-spec целиком **не глотали**: если стоит - зовём; если нет - свои шаблоны. Проще откатить и не плодить «ещё один скилл ради скилла».

## Как я этим пользуюсь

Это не брошюра продукта. Так я реально работаю. `project-orchestra` как раз про этот порядок: один вход, волны, файл «можно делать», проверка двумя моделями. Без среды, где несколько агентов сидят рядом, половина смысла - теория.

### Orca - где это удобно крутить

Я использую [**Orca**](https://onorca.dev) ([репозиторий: stablyai/orca](https://github.com/stablyai/orca)). Они называют это ADE - среда **под агентов**, а не «ещё один редактор кода». Смысл с их сайта близок к тому, что мне нужно: обычные IDE делали для человека; ADE - для человека **и** агентов (отдельные рабочие копии, терминалы, браузер, CLI в одном окне). Плюс *bring your own agent / subscription*: Claude Code, Codex, OpenCode, Grok и другие CLI рядом, на **своих** подписках.

Зачем мне это по делу:

1. **Несколько агентов одновременно без цирка.** Когда нужно две разные проверки, открываю два terminal в Orca, указываю какой агент и какая модель, жду ответы **файлами на диске**. Два голых `opencode run` у меня лочили базу - с этим уже наелся.
2. **Проекты и рабочие копии в одном окне.** Не размазываю пятнадцать вкладок iTerm «где там DeepSeek».
3. **Команда `orca`.** Ведущий агент сам может создать terminal и отправить туда задачу - удобно, когда скилл говорит: два исполнителя, у каждого своя роль (один сверяет факты, другой ищет, где всё сломается).
4. **Разные CLI в одной оболочке.** Grok, Claude Code, OpenCode - не прыгаю каждый час между приложениями.

**Что нравится:**

- реально можно держать несколько агентов и не терять, кто чем занят;
- open source (MIT), сайт и релизы открыты;
- есть приложение на телефон - пока не использовал;
- не навязывает «только их модель» - тащишь свои CLI и ключи.

**Что бесит / где трение:**

- это ещё один слой поверх обычного терминала. Если агент у тебя один и навсегда - Orca может быть лишним;
- продукт быстро меняется, привычки UI плавают;
- к отдельным рабочим копиям и «ручкам» terminal’ов надо привыкнуть;
- Orca **не заменяет** дисциплину (файл «можно делать», STATUS, HANDOFF). Она только **перевозит** эту дисциплину.

Одной фразой: скилл говорит *что* делать. Orca - *где* нескольким агентам жить, чтобы они не бились лбами.

- сайт: [onorca.dev](https://onorca.dev)
- код: [github.com/stablyai/orca](https://github.com/stablyai/orca)

### Какие модели я кручу

На одной модели не сижу. Сейчас так:

| Где | Что | Зачем |
|-----|-----|-------|
| **Grok 4.5** (родной build) | Часто ведущий / длинные сессии | Быстрый цикл «подумал - написал - проверил» |
| **Claude Code** | Ведущий на **GLM 5.2** (и когда удобна именно оболочка Claude Code) | Другая голова и другой UX; удобно вести план и исполнителей |
| **OpenCode** | **GLM 5.2**, **DeepSeek V4 Pro** (и другие по задаче) | Исполнители: сверка по чеклисту, «а что сломается», выполнение |

Почему не «везде Claude» и не «везде Grok»:

- **Разные семьи** на двойной проверке. DeepSeek и GLM у меня часто дополняют друг друга: один держит чеклист и доказательства, другой ломает сценарии «а если в прод».
- **Ведущий** - обычно Claude Code (часто GLM 5.2) **и/или** Grok 4.5 в родном build. Смотрю, где сегодня меньше трения. Не религия.
- **Личные инструкции агентов** (роли, AGENTS, persona) **наружу не отдаю**. В публичном скилле - только общий каркас: режимы, stamp, кто что пишет, рецепт Orca. Кухню промптов оставляю себе.

Коротко: *несколько оболочек + несколько семейств моделей + Orca как стол, за которым они не сталкиваются*.

## Как устроены папки

```
проект (весь продукт)
├── AGENTS, HANDOFF, MEMORY, SPEC…   - общий «мозг»
├── optimize/                        - тема работ
│   └── waves/2026-07-15-…/          - один кусок: план, «можно делать», отчёт
└── landing/
    └── waves/…
```


| Слой | Простыми словами | Режим |
|------|------------------|-------|
| **Корень проекта** | Весь продукт, общие правила и память | `full` или `bootstrap-lite` |
| **Тема** | Подпапка по смыслу (не обязательно отдельный git) | `workstream-new` |
| **Волна** | Один законченный кусок работы | `wave`, потом ревью, потом execute |

## Что будет при запуске

1. Агент **смотрит** текущую папку (git не обязателен - хватает файлов).
2. Коротко пишет: что увидел и что предлагает.
3. Если неясно - **до пяти вопросов** (полный «офис» или четыре файла, где корень, одна модель или несколько, это прод или нет, какая тема).
4. Ты подтверждаешь режим.
5. Появляются файлы и прогоняются проверки. «Готово» - когда есть артефакты и зелёные проверки, а не «ну вроде».

Можно сразу: «новая тема optimize» или «только четыре файла» - вопросов меньше.

## Режимы


| Режим | Одной фразой |
|-------|--------------|
| `full` | Полный «офис» для нескольких агентов |
| `bootstrap-lite` | Только четыре файла: AGENTS, HANDOFF, MEMORY, gitignore |
| `workstream-new` | Новая тема-папка под уже существующим проектом |
| `wave` | Набросать план волны (wave-spec, если есть, или шаблоны) |
| `raeh-review` | Ревью до «можно делать» (stamp + hash) |
| `raeh-execute` | Сделать утверждённое и написать отчёт |
| `install-dialects` | Только шпаргалки, как писать промпты ролям |
| `extend` | Докрутить уже существующее: роли, waves, уборка |

## Где помогает, а где нет


| Ситуация | Поможет? | Комментарий |
|----------|----------|-------------|
| Несколько агентов, непонятно кто главный | Да | `full` или `extend` |
| Проект есть, нужна новая длинная тема | Да | тема, потом волна |
| План, проверка двумя моделями, работа с отчётом | Да | stamp + две модели |
| Пустая папка, «дай файлы для агента» | Да, короткий путь | `bootstrap-lite` |
| Нужна SEO-методика | Один не потянет | Сначала офис, потом SEO-скиллы |
| Одна правка, уже ок | Нет | Лишнее |
| «Сам задеплой в прод без меня» | Нет | Прод остаётся на человеке |

## Установка

### OpenCode

```bash
git clone git@github.com:dimkurilo/opencode-skills.git ~/Projects/opencode-skills
ln -sfn ~/Projects/opencode-skills/skills/project-orchestra \
  ~/.config/opencode/skills/project-orchestra
```

### Grok

```bash
ln -sfn ~/Projects/opencode-skills/skills/project-orchestra \
  ~/.grok/skills/project-orchestra
```

### Скрипты, если руками

```bash
# четыре файла
bash …/scripts/install_bootstrap_lite.sh /path/to/project "name"

# полный «офис»
bash …/scripts/install_project_os.sh /path/to/project "name"

# тема (у корня уже должен быть AGENTS.md)
bash …/scripts/install_workstream.sh /path/to/project "theme-slug"
```

## Правила, которые реально важны

1. «Можно делать» пишется в **файле-штампе**. Сообщение в чате «ок, делай» само по себе не считается.
2. План волны можно в `.md` или в `.xml` - для hash оба ок.
3. Тот, кто **делает** работу, не правит STATUS, MEMORY и HANDOFF - это зона ведущего.
4. Если нужны две разные модели, а есть только одна семья - так и пиши: `DEGRADED_DUAL`.
5. Два агента параллельно - **два terminal в Orca**, не два голых `opencode run`.
6. В пакет скилла не кладём hostname’ы клиентов и SEO-доктрину.

Полный протокол для агента: [`SKILL.md`](SKILL.md). Справки: `references/`.

## Чего лучше не делать

- Ставить полный «офис», когда просили четыре файла.
- Делать тему-папку, если в корне ещё нет AGENTS.
- Считать чат «договорились» достаточным, чтобы начать execute.
- Два одинаковых ревью одной моделью и называть это «проверкой двумя».
- Тащить в скилл SEO-рецепты и адреса чужих сайтов.
