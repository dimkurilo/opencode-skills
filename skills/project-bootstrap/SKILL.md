# project-bootstrap

Создаёт агентскую инфраструктуру для нового проекта.
Следует стандарту Agent Playbook v0.0.5 (https://agents.md).

---

## Workflow

### 0. Принять задачу

Прочитай описание задачи пользователя. Это может быть:
- Свободный текст («надо бекапить 3 сервера...»)
- Ссылка на файл со спецификацией
- Голосовое/текстовое описание любой степени детализации

Не требуй формальной спецификации. Работай с тем, что дано.

### 1. Анализ: «тёмные пятна»

Определи, что нужно уточнить ИЗВНЕ (не из памяти модели):

**1a. Внешние инструменты и CLI**
Если в задаче упоминаются конкретные тулы (b2, rsync, pg_dump, docker, aws-cli, yt-dlp, gh, glab и т.д.):
- Сделай webfetch/websearch по запросам вида:
  - `"<tool> best practices 2025 2026"`
  - `"<tool> common pitfalls <task type>"`
  - `"<tool> CLI reference flags"`
- Собери актуальный синтаксис, флаги, ограничения.
- Занеси в MEMORY.md раздела «Инструменты и команды».

**1b. Существующие проекты**
Проверь, есть ли у пользователя похожие проекты. Прочитай их `.agents/memory/MEMORY.md`:
- Учётные данные (без паролей, только имена/пути)
- Известные грабли
- Рабочие процедуры
- Переиспользуй найденное в новом проекте.

**1c. API и протоколы**
Если задача касается API (Backblaze B2, S3, REST, GraphQL):
- Сделай webfetch документации: актуальные эндпоинты, аутентификация, лимиты.
- Занеси в MEMORY.md.

### 2. Анализ: определение структуры

На основе задачи определи, какие модули `.agents/` нужны:

| Модуль | Когда нужен | Признаки в задаче |
|--------|-----------|-------------------|
| `memory/MEMORY.md` | **Всегда** | — |
| `rules/general.md` | **Всегда** | — |
| `rules/*.md` | Есть повторяемые операции | «всегда проверяй», «формат такой-то», «безопасность» |
| `agents/` | Есть роли-исполнители | «один делает X, другой Y», «@роль» |
| `commands/` | Есть повторяемые команды | «/status», «/deploy», «/backup» |
| `skills/` | Есть workflow из нескольких шагов | «сначала A, потом B, затем C» |
| `skills/*/scripts/` | Нужны детерминированные операции | CLI-цепочки, парсинг, конвертация |
| `docs/` | Нужна формальная спецификация | Пользователь передал SPEC/PRD |

**Правило:** создавай только то, что задача реально требует. Не раздувай.

### 3. Генерация: создание файлов

#### 3a. `AGENTS.md` (корень проекта)

Создай главный манифест. Используй шаблон из `assets/templates/AGENTS.md.tmpl`.
Обязательные секции:
- **Описание проекта** — извлеки из задачи пользователя (1 абзац)
- **Архитектура** — древовидная схема `.agents/`
- **Loaded Context** — таблица со всеми созданными файлами, Auto-load/on-demand
- **Критические правила** — 3-5 самых важных (язык, верификация, запрет галлюцинаций)
- **Протокол работы** — основные сценарии (если применимо)
- **Setup commands** — команды для работы с проектом (если применимо)

#### 3b. `.agents/memory/MEMORY.md`

Используй шаблон из `assets/templates/MEMORY.md.tmpl`.
Заполни секции:
- **CONFIRMED_FACTS** — инвентаризация: серверы, API, файлы, БД, инструменты (из фазы 1)
- **Решения** — пока пусто, шаблон для будущих записей
- **Инструменты и команды** — CLI-синтаксис из webfetch (фаза 1a)
- **Известные ограничения** — лимиты API, грабли из webfetch

**ВАЖНО:** никаких паролей, токенов, ключей в MEMORY.md. Только имена, пути, флаги.

#### 3c. `.agents/rules/general.md`

Используй шаблон из `assets/templates/general-rule.md.tmpl`.
Базовые правила для любого проекта:
- Язык общения (русский, если не указано иное)
- Прямой стиль, без лести
- Перед изменениями — прочитать релевантные rules и memory
- Верификация после каждого изменения
- Одна задача за раз
- Failure Packet при 2+ ошибках
- Не галлюцинировать данные

#### 3d. Дополнительные модули

**Если нужны agents/** — создай файлы `.agents/agents/<role>.md`:
Используй шаблон `assets/templates/agent-persona.md.tmpl`.
Секции: Identity, Responsibilities, Workflow, Boundaries, Output Format.

**Если нужны commands/** — создай `.agents/commands/<name>.md`:
Используй шаблон `assets/templates/command.md.tmpl`.
Секции: Usage, Behavior, Output Format.

**Если нужны skills/** — создай директорию `.agents/skills/<name>/`:
- `SKILL.md` — front matter + Workflow + Boundaries
- `WORKFLOW.md` — пошаговый алгоритм
- `scripts/` — Python/Bash скрипты для детерминированных операций

**Если нужны docs/** — создай `docs/YYYY-MM-DD-task-name/`:
- На основе переданной пользователем спецификации

### 4. Верификация

После генерации:
1. Проверь, что все созданные файлы читаются (read каждого файла)
2. Проверь, что `AGENTS.md` содержит Loaded Context для всех созданных модулей
3. Проверь валидность YAML front matter во всех skills, rules, agents, commands
4. Выведи сводку: что создано, почему, что можно добавить позже

## Decision Framework: MCP vs CLI vs Script

При анализе задачи применяй эту логику:

| Ситуация | Решение | Пример |
|----------|---------|--------|
| Частая/защищённая операция | MCP (упомянуть в AGENTS.md) | Confluence, Jira, GitHub API |
| Редкая/сложная операция | CLI-команда (в commands/) | `gh pr create`, `aws s3 sync` |
| Цепочка CLI+MCP | Скрипт в `skills/*/scripts/` | yt-dlp → метаданные → S3 |
| Повторяемый workflow | Skill с WORKFLOW.md | «бекап сервера» = 5 шагов |

---

## Что НЕ делать

- **НЕ** задавать больше 2 уточняющих вопросов. Если информации недостаточно —
  создай минимальную структуру (AGENTS.md + memory + rules) и отметь, что можно добавить позже.
- **НЕ** спрашивать «нужны ли тебе agents/?» — определи по задаче.
- **НЕ** галлюцинировать факты. Если чего-то нет в спеке или webfetch — оставь `<!-- TODO -->` в MEMORY.md.
- **НЕ** создавать избыточные модули. Одна задача = один скрипт → не нужен agents/.
- **НЕ** полагаться на память модели для CLI-синтаксиса — всегда проверяй через webfetch.
- **НЕ** хранить пароли, токены, ключи в любых файлах `.agents/`.
- **НЕ** требовать от пользователя формальной спецификации — работай с тем, что дано.

## Пример

**Пользователь:** «Надо настроить бекапы 3 серверов. Сервер 1 (Contabo) — скачать qcow2 образ на Backblaze B2, не через ноут. Сервер 2 и 3 — данные бекапить через rukbu agent на GoClaw. Доступ: SSH-ключи в ~/.ssh/config.»

**Скилл:**
1. webfetch: `"backblaze b2 cli backup qcow2 best practices"`, `"b2 upload large files flags"`
2. read: `contabo-goclaw/.agents/memory/MEMORY.md` (SSH, конфиги)
3. read: `contabo-goclaw/.agents/rules/operations.md` (процедуры)
4. Создаёт:
   - `AGENTS.md` — проект «Бекапы серверов», Loaded Context
   - `memory/MEMORY.md` — инвентаризация 3 серверов + Backblaze, b2 CLI синтаксис
   - `rules/backup-procedures.md` — алгоритмы: qcow2 бекап, данные бекап
   - `commands/backup-status.md` — `/backup-status`
   - `skills/backup-qcow2/` + `scripts/backup-image.sh`
   - `skills/backup-regular/` + `scripts/backup-data.sh`
5. Вывод: «Структура готова. Серверы занесены в MEMORY.md. AGENTS.md содержит протокол. Начинаем?»
