# Variant E — Структура AGENTS.md

Архитектурный паттерн для главного файла инструкций агента. Решает проблему «Lost in the Middle»: критические правила теряются в середине документа, модель их игнорирует.

---

## Проблема

В стандартном AGENTS.md архитектура проекта — в начале (primacy-зона), критические правила — в середине или раскиданы по файлу. «Lost in the Middle» (Liu et al., 2023): правила на позициях 30-80% файла теряются моделью. Практическое решение проверено на DeepSeek и GLM: правила в начале (primacy) + дубликат в конце (recency) — неизбежны.

## Решение: Variant E

```
┌─ ЗАГОЛОВОК
├─ 🔴 ПРЕАМБУЛА: N ЖЕЛЕЗНЫХ ПРАВИЛ (IF-THEN, primacy-зона)
├─ 📋 ЧЕКЛИСТ ПЕРЕД ДЕЙСТВИЕМ
├─ 📐 ПРОТОКОЛ (сжатый, command-first)
├─ §1 Известные проблемы / Gotcha's
├─ §2 Failure Packet
├─ §3 Иерархия инструкций
├─ §4 Архитектура проекта + Loaded Context (ДНО)
├─ §5 Исторические сессии (ДНО, только ссылки на SESSION_HANDOFF.md)
└─ CLOSING ANCHORS (recency-зона: правила продублированы в конце)
```

### Принцип

1. **Primacy (начало)** — железные правила IF-THEN + чеклист. Агент видит сразу при загрузке.
2. **Middle (середина)** — справочная информация: gotcha's, failure packet, иерархия.
3. **Recency (конец)** — архитектура + исторические сессии (дно) + CLOSING ANCHORS (дубликат критических правил).

### Почему работает

- Правила в преамбуле = primacy effect (первые строки запоминаются).
- Правила в CLOSING ANCHORS = recency effect (последние строки имеют максимальный вес в DeepSeek V4).
- Правила неизбежны: агент не может их пропустить, где бы ни находился в документе.
- Архитектура на дне: справочная информация, не требует запоминания.

### Когда применять

| Тип проекта | Имя варианта | Что генерировать |
|------------|-------------|-----------------|
| ops-сервер (Docker, SSH, продакшен-риски) | `variant-e-full` | Полный набор: ПРЕАМБУЛА + ЧЕКЛИСТ + ПРОТОКОЛ + §1 Gotchas + §2 Failure Packet + §3 Иерархия + §4 Архитектура + Loaded Context (L1/L2/L3 таблицы) + §5 Сессии + CLOSING ANCHORS |
| код-проект (py/ts/js, тесты, CI/CD) | `variant-e-grace` | То же что full + GRACE-якоря в кодовых файлах |
| агентский (агенты, скиллы, промпты) | `variant-e-model` | То же что full + модельно-специфичные CLOSING ANCHORS (см. `references/model-profiles.md`) |
| контент/бизнес (статьи, Excel) | `lightweight` | Только: ПРЕАМБУЛА + §4 Архитектура (без таблиц Loaded Context) + §5 Сессии + @rule в конце. **НЕ включать:** ЧЕКЛИСТ, ПРОТОКОЛ, §1-§3, Loaded Context таблицы |
| неопределённый (<3 файлов) | `base` | Минимальный: ПРЕАМБУЛА (1-2 базовых правила) + §4 Архитектура (простая) + §5 Сессии. Предложить доработать до нужного варианта |

**Правила для lightweight (`lightweight`):**
- Опустить: ЧЕКЛИСТ, ПРОТОКОЛ, §1 Gotchas, §2 Failure Packet, §3 Иерархия.
- Опустить: Loaded Context таблицы (L1/L2/L3).
- Сохранить: ПРЕАМБУЛА, §4 Архитектура (без таблиц), §5 Сессии, @rule в конце.
- ПРЕАМБУЛА содержит только базовые правила (privacy, data, anti-hallucination).

**Правила для base (`base`):**
- Опустить: всё кроме ПРЕАМБУЛЫ, §4 Архитектура (простая), §5 Сессии.
- ПРЕАМБУЛА: 1-2 базовых правила.
- Добавить: `<!-- TODO: доработать до variant-e-full / variant-e-grace / lightweight -->` после closing anchors.

### Адаптация под модель

- **DeepSeek V4:** CLOSING ANCHORS с recency-оптимизацией (правила продублированы в конце).
- **GLM 5+:** указание «этот файл должен быть загружен перед SESSION_HANDOFF.md» (state-continuity).
- **Универсальный:** оба варианта.

## Примеры преамбул (few-shot, адаптируются под тип проекта)

### Универсальные правила (в любом проекте)

```markdown
## 🔴 ПРЕАМБУЛА: 3 ЖЕЛЕЗНЫХ ПРАВИЛА

➡ Сделал изменение?
   → НЕМЕДЛЕННО проверь результат командой. «Готово» = проверяемый результат, не ощущение.

➡ Получил 2 одинаковые ошибки подряд?
   → СТОП. Failure Packet. Смени подход. Не пытайся «починить» тем же методом.

➡ Видишь факт/цифру/команду не из источника?
   → НЕ галлюцинируй. Не знаешь → спроси или `<!-- TODO -->`.
```

### Ops-проект (Docker, серверы, продакшен-риски)

```markdown
## 🔴 ПРЕАМБУЛА: ЖЕЛЕЗНЫЕ ПРАВИЛА (нарушение = простой)

➡ Изменяешь КОНФИГИ на сервере?
   → СНАЧАЛА backup: mkdir -p "backups/$(date +%Y-%m-%d)-task/files"
   → СНАЧАЛА pre-check: uptime, free -h, df -h, docker ps, curl -sI https://...

➡ Использовал cache:clear (php artisan)?
   → НЕМЕДЛЕННО: docker restart <ct> + sleep 3 + curl -sI https://...
```

### Код-проект (исходники, тесты, CI/CD)

```markdown
## 🔴 ПРЕАМБУЛА: ЖЕЛЕЗНЫЕ ПРАВИЛА

➡ Деплоишь в продакшен?
   → СНАЧАЛА: тесты проходят, staging проверен, backup сделан.

➡ Меняешь API / контракты?
   → СНАЧАЛА: проверь контракт в staging. НЕ деплой в пятницу.
```

### Контент/бизнес-проект (статьи, данные, аналитика)

```markdown
## 🔴 ПРЕАМБУЛА: БАЗОВЫЕ ПРАВИЛА

➡ Работаешь с персональными данными?
   → НЕ загружай реальные ФИО/телефоны/email. Обезличь перед обработкой.

➡ Генерируешь текст/выводы?
   → Только из источников. Каждая цифра должна иметь источник. НЕ придумывай.
```

## Пример CLOSING ANCHORS

```markdown
<!-- CLOSING ANCHORS — правила в конце имеют максимальный вес (DeepSeek recency effect) -->
<!-- @rule id="F-01" priority="critical" -->
➡ Изменяешь КОНФИГИ? → СНАЧАЛА backup.
<!-- @rule-end -->
```
