# Workflow Patterns — Каталог архитектурных паттернов

Справочник для выбора архитектуры при создании сложных skills в `.agents/skills/`.
Модель сама выбирает подходящий паттерн или комбинацию паттернов под задачу.

---

| Паттерн | Когда применять | Признаки в задаче |
|---------|----------------|-------------------|
| **Classify-and-Act** | Разные типы операций требуют разных обработчиков | «для X сделай A, для Y сделай B», «в зависимости от типа» |
| **Fan-out-and-Synthesize** | Много однотипных элементов → параллельно → склеить результат | «проверь 50+ элементов», «для каждого файла», «по всем серверам» |
| **Adversarial Verification** | Проверка своей же работы другим агентом (bias protection) | «проверь», «отыщи ошибки», «аудит», «соответствует ли» |
| **Generate-and-Filter** | Много идей/вариантов → отфильтровать по критериям | «придумай N вариантов, выбери лучшие», «сгенерируй и отбери» |
| **Tournament** | Попарное сравнение вариантов → победитель | «выбери лучший», «сравни», «ранжируй», «кто победит» |
| **Loop Until Done** | Неизвестный объём работы → повторять до условия остановки | «пока не найдёшь», «исследуй пока...», «до первопричины» |
| **Multi-Model Cross-Validation** | Один анализ → 2-3 разные модели → синтез результатов | «разные ракурсы», «нужна объективность», «проверить разными моделями» |
| **Review→Fix Loop** | Сделал → получил фидбек → исправил → повторить до одобрения | «итаративно», «ревью», «проверить и исправить», «сделать до утверждения» |
| **Data Sanitization Pipeline** | Raw данные → обезличить → проверить → обработать → восстановить | «NDA», «обезличивание», «персональные данные», «чувствительные данные» |

---

## Примеры комбинаций

**Диагностика (root cause):**
Fan-out (собрать данные) → Loop Until Done (проверять гипотезы) → Adversarial Verification (подтвердить первопричину)

**Сортировка большого списка:**
Fan-out (разбить на батчи) → Tournament (попарное сравнение) → Fan-out (параллельная обработка топ-N)

**Контент-генерация с качеством:**
Generate-and-Filter (нагенерировать варианты) → Adversarial Verification (проверить качество) → Tournament (выбрать лучший)

**Анализ чувствительных данных:**
Data Sanitization (обезличить → проверить) → Multi-Model Cross-Validation (2-3 модели) → Review→Fix Loop (итеративно довести до качества)

**Аргументация для стейкхолдера:**
Multi-Model Cross-Validation (разные ракурсы) → Review→Fix Loop (итерации с ревьюером) → Adversarial Verification (финальная проверка)

---

## Когда НЕ применять

- Простая задача из 1-2 шагов — не нужен паттерн, делай прямо
- Тривиальная проверка — не нужен adversarial, самопроверки достаточно
- Меньше 5 элементов — не нужен fan-out, последовательно быстрее
